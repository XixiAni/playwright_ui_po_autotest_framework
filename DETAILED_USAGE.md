# 框架详细使用说明

## 一、框架核心设计原理

### 1. 浏览器选型说明
#### 背景问题
官方 Playwright 需通过 `playwright install` 下载定制版 Chromium，但 Windows 设备（尤其带第三方安全防护的设备）会拦截浏览器内核 dll 文件的自动解压写入，持续抛出 `EPERM: operation not permitted` 权限错误，无法完成安装。

#### 解决方案
框架直接调用系统预装的 Microsoft Edge 浏览器：
- Edge 基于 Chromium 内核开发，与 Playwright 官方 API 100% 兼容，所有元素操作、断言、等待语法完全通用
- 无需下载额外浏览器包，跳过解压写入环节，彻底规避权限拦截问题
- 支持两种启动方式：`executable_path` 指定本地路径、`channel="msedge"` 自动识别系统路径

#### 启动兜底机制
- 1. 优先读取 `config.yaml` 中 `browser.executable_path` 指定的本地路径
​
- 2. 若路径为空或文件不存在，自动降级为 `channel="msedge"` 模式，自动识别系统Edge安装位置
​
- 3. 两种模式API 100%兼容，切换无感知，运行日志会明确标记当前使用的启动方式

### 2. 夹具联动机制
框架采用 Pytest Fixture 依赖注入机制，分层管理浏览器与页面生命周期，无需手动编写启动/关闭代码。

#### 分层设计
#### 分层设计
| 夹具名  | 作用域  | 生命周期                 | 核心作用                                                     |
|---------|---------|--------------------------|--------------------------------------------------------------|
| browser | session | 整个测试会话仅执行1次    | 全局启动 Edge 浏览器，所有用例共享同一浏览器进程             |
| page    | function| 每个测试用例执行1次      | 为单条用例新建独立标签页，执行完自动关闭，隔离用例数据       |


#### 单条用例完整执行时序
1. Pytest 扫描测试用例，识别到用例依赖 `page` 夹具
2. 逐层向上查找依赖：`page` 依赖 `browser`，优先执行 `browser` 夹具
3. `browser` 前置逻辑：启动 Playwright 驱动，拉起 Edge 浏览器
4. `page` 前置逻辑：基于全局浏览器新建空白标签页
5. 执行测试用例业务代码与断言
6. `page` 后置逻辑：关闭当前标签页，清理用例缓存与Cookie
7. 全部用例执行完毕，`browser` 后置逻辑：关闭浏览器进程，释放资源

### 3. 鲁棒性体系设计说明
针对线上页面动态渲染、弹窗遮挡、元素隐藏等不稳定场景，框架搭建了五层容错体系：

#### 3.1 浏览器环境层
- 无痕模式启动，隔离本地缓存与历史配置，保证每次测试环境纯净一致
- 注入JS覆盖 navigator.webdriver 属性，去除自动化内核标识，降低站点风控识别概率
- 固定桌面端视口尺寸，并监听窗口变化强制锁定，避免页面自适应切换移动端布局

#### 3.2 页面兼容层
- 多类型弹窗自动关闭：循环匹配多种常见弹窗关闭选择器，不存在则跳过，避免浮层遮挡目标元素
- 分级等待机制：支持 DOM挂载/元素可见 两种等待状态，适配元素先挂载后渲染的动态场景

#### 3.3 操作容错层（核心）
采用「原生严格单次 + 原生不严格单次 + JS注入兜底」的三层线性降级策略，解决元素动态隐藏、渲染延迟导致的操作超时问题，废弃原有的循环重试模式，减少无效等待耗时。
- 第一层：短超时等待元素可见，调用 Playwright 原生 fill/click API，贴近真实用户操作，用于正常渲染元素快速通路
- 第二层：仅等待DOM挂载，通过force参数绕过可见性校验，执行原生操作，适配DOM已挂载但视觉隐藏的元素（如动态渲染按钮、CSS隐藏输入框）
- 第三层：原生操作全部失败时，自动降级为 JS 注入，直接操作DOM赋值/触发点击事件，完全绕过Playwright内置校验，作为最终兜底
- 安全加固：JS兜底统一采用官方参数传参写法，自动转义特殊字符，杜绝字符串拼接的语法风险与注入漏洞
- 关键细节：JS赋值后必须手动派发 input 事件，保证页面前端交互逻辑正常触发
> 说明：元素存在但不可见导致的超时属于预期内容错场景，框架会自动降级执行，不影响用例最终通过，不属于代码故障。
> 补充：Locator实例化场景下的JS兜底兼容设计
  全框架切换为Locator对象后，Playwright未开放反向提取内部选择器的公开API，直接传入Locator会导致JS注入逻辑无法获取目标元素选择器。
  框架采用「原始字符串预缓存」方案解决：

1. 业务页面初始化时，除生成Locator实例外，同步从YAML读取原始选择器字符串存入私有缓存
​
2.  `safe_fill` / `safe_click`  方法新增  `raw_sel`  入参，业务层调用时传入缓存的原始字符串
​
3. JS兜底分支优先使用手动传入的  `raw_sel` ，完全不依赖Locator对象的内部属性，保证降级逻辑永久生效

#### 3.4 等待优化层
- 淘汰固定时长硬等待（wait_for_timeout），统一使用基于元素状态的智能等待（`wait_for_selecto`r）
- 智能等待原理：循环轮询检测元素状态，满足条件立即放行，timeout仅为兜底上限，兼顾效率与稳定性
- 状态说明：
  - attached：仅要求DOM节点挂载到页面，不要求视觉可见
  - visible：要求DOM存在且页面肉眼可见，是原生操作的前置校验标准

#### 3.5 断言增强层
- 缩小断言范围：锚定业务级元素（如单条搜索结果），而非全局body，减少无效内容干扰
- 消除大小写差异：统一转小写后匹配，规避页面文字大小写不统一的问题
- 多关键词遍历匹配：使用any()校验多个核心关键词，任意匹配即通过，降低排版变化对断言的影响

## 二、PO页面对象模式设计

### 1. 分层设计思想
采用「基类通用能力 + 页面业务逻辑」的两级分层，实现页面细节与测试用例解耦：
- 公共层：所有页面通用的操作统一收敛到 BasePage，一处修改全页面生效
- 业务层：每个页面对应一个 Page 类，仅维护自身定位符与业务流程
- 用例层：只调用业务方法，不出现任何元素选择器、底层API，页面改版不影响用例代码

### 2. BasePage 通用基类定位
 
作为所有业务页面的父类，收敛全项目通用UI操作与容错能力，完整方法清单与使用规范见第七章7.1.1。
 
### 3. 业务页面类（SearchPage）定位
 
继承 BasePage 复用通用能力，仅维护自身业务流程与元素定位，完整字段与方法说明见第七章7.2.1。

### 4. 核心扩展能力：下标索引透传
- 基层 `get_element_text` 预留 `index` 参数，可读取匹配列表中任意序号的元素
- 业务层 `get_indexed_result_text` 对外透传 `index` 参数，默认值为 0，向下兼容原有单条结果读取逻辑
- 使用示例：
```python
# 获取第一条结果（默认）
baidu_page.get_indexed_result_text()
# 获取第三条结果
baidu_page.get_indexed_result_text(index=2)
```
 
### 5. 环境隔离原理
 
- PO实例隔离：多个PO类实例是独立的Python对象，但传入同一份 page 引用时，共享同一个浏览器标签页的缓存与Cookie
- 真正隔离：依靠  `scope="function"`  的 page 夹具，每条用例新建独立标签页，用例间数据完全隔离

## 三、用例失败自动重试机制
 
### 1. 设计目标与适用场景
 
针对网络波动、页面偶发渲染、接口超时等非必现故障，自动重跑用例，降低误报率，提升自动化执行稳定性。仅对业务执行阶段（call）失败触发重试，夹具初始化/销毁异常不重试。
 
### 2. 三层优先级规则
 
优先级从高到低，高优先级配置会覆盖低优先级：
 
1. 单用例装饰器： `@pytest.mark.flaky(reruns=3, reruns_delay=2)` ，单条用例自定义重试规则
2. 命令行参数： `pytest --reruns 2 --reruns-delay 1` ，本次运行全局生效
3. YAML全局配置： `config.yaml`  中  `rerun` 节点，默认全局规则
 
### 3. 配置说明
 
文件路径： `config/config.yaml` 
 
```yaml
  
rerun:
  enable: true          # 全局重试总开关
  max_times: 2         # 最大重试次数（首次失败后额外重跑的次数）
  delay: 1             # 每次重试间隔时间，单位秒
``` 
 
- 配置字段缺失时自动兜底： `max_times`  默认1次， `delay`  默认1秒
-  `enable: false`  或配置读取失败时，全局关闭重试，不阻断用例执行
 
### 4. 底层实现原理
 
通过  `pytest_configure`  钩子在测试会话启动前注入全局重试参数，核心逻辑如下：
 
#### 4.1 钩子触发时机
 
pytest 读取完命令行参数、加载所有插件后，收集用例前执行，全局仅执行1次，是注入全局配置的标准时机。
 
#### 4.2 关键判断逻辑踩坑记录
 
- 错误写法1： `config.option.reruns_set`  属性不存在，会抛出  `AttributeError` ，不可使用
- 错误写法2： `if config.getoption("reruns") == 0`  无法区分「无传参默认0」和「用户手动--reruns 0显式关闭」，会错误覆盖用户输入
- 正确写法： `config.getoption("reruns", None)` ，通过第二个参数 None 区分用户是否手动传参
- 返回 None：用户未输入，加载YAML配置
- 返回数字（含0）：用户手动指定，跳过YAML配置

#### 4.3 代码语义化优化说明

1. 兜底字典语义精准：仅在YAML读取异常的 `except` 分支赋值关闭重试的兜底字典，外层仅做空字典占位，符合“异常兜底”语义
​
2. 日志分层可追溯：拆分三种传参场景独立日志，清晰区分参数来源
​
3. 属性说明： `config.option.reruns`  /  `config.option.reruns_delay`  是  `pytest-rerunfailures`  插件内置属性；命令行传参会由pytest自动写入该属性，代码仅在用户未传参时做覆盖赋值
 
### 5. 单用例重试装饰器用法
 
```python
  
import pytest
@pytest.mark.flaky(reruns=3, reruns_delay=2)
def test_search_function(page):
    # 用例逻辑
```

-  `reruns` ：重试次数
-  `reruns_delay` ：重试间隔，单位秒
- 优先级高于全局配置，适合偶发故障特别高的单条用例
 
### 6. 与现有框架的联动逻辑
 
- 页面夹具联动：每次重试都会重新执行 function 级 page 夹具，新建独立标签页，无缓存污染
- 失败截图联动：每一轮失败都会触发自动截图钩子，生成独立截图文件
- Allure报告联动：每一轮执行结果独立记录，重试过程可追溯

### 7. 兼容边界说明
 
- ✅ 完全兼容：function级page夹具用例，每次重试重建独立页面，无状态污染
​
- ⚠️ 有限兼容：class级共享夹具用例，重试复用同一页面实例，页面状态持续累积，可能放大数据污染，不推荐搭配重试使用
​
- ❌ 不触发场景：setup/teardown阶段异常、夹具初始化失败，不触发重试

### 8. 验证方法
 
1. 人为制造断言失败，执行用例
2. 观察控制台出现  `RERUN`  标识，次数与配置一致
3. 最终统计行显示  `x rerun` ，匹配配置的重试次数
4. 查看截图目录，失败轮次对应独立截图文件

## 四、多环境YAML切换体系
 
### 1. 设计思想：公共配置与环境配置分离
 
将配置拆分为「全环境通用公共配置」和「单环境差异化配置」，同一业务系统在不同部署环境（dev/test/prod）仅域名、少量UI差异，无需重复编写全量配置。
 
- 公共配置：浏览器参数、超时、通用定位器等所有环境通用的参数
- 环境配置：页面URL、测试账号、差异化定位器等环境独有参数
 
### 2. 目录结构说明
 
```text
  
config/
├── config.yaml      # 全局框架公共配置（浏览器、超时、报告、重试）
├── locators.yaml    # 全环境公共元素定位器
└── env/             # 多环境差异化配置目录
    ├── dev.yaml     # 开发环境
    ├── test.yaml    # 测试环境
    └── prod.yaml    # 生产环境
``` 
 
### 3. 环境切换命令与参数解析
 
#### 3.1 切换命令
 
```bash
  
# 默认开发环境
pytest tests/test_search.py -vs

# 指定测试环境
pytest tests/test_search.py -vs --env=test

# 指定生产环境
pytest tests/test_search.py -vs --env=prod
``` 
 
#### 3.2 命令行参数注册原理
 
通过  `pytest_addoption`  钩子注册自定义  `--env`  参数，参数说明：
 
-  `--env` ：参数名，命令行输入使用
-  `action="store"` ：接收一个字符串值，是最常用的参数模式
-  `default="dev"` ：不传参数时的默认值，保证向下兼容
-  `choices=["dev", "test", "prod"]` ：合法值白名单，输入其他值直接报错拦截
-  `help="..."` ： `pytest -h`  帮助文档展示的说明文字
 
### 4. 全局环境初始化机制
 
#### 4.1 初始化时机
 
在  `pytest_configure`  钩子最前端执行环境初始化，早于所有配置读取、夹具初始化，保证全流程环境统一。
 
#### 4.2 环境夹具说明
 
```python
  
@pytest.fixture(scope="session")
def env(request):
    env_name = request.config.getoption("--env")
    return env_name
``` 
 
-  `request` ：pytest 内置夹具，可读取本次运行全部命令行、配置信息
- 作用：测试用例或其他夹具如需获取当前环境名，可注入该夹具
- 全局环境标识存储在  `YamlReader._current_env`  静态变量中，业务代码无需手动传参
 
### 5. 环境定位器覆盖原理
 
#### 5.1 核心规则
 
- 公共  `locators.yaml`  作为基准，存放所有环境通用的定位器
- 各环境 yaml 中仅存放与公共版本有差异的定位器，不需要全量复制
- 读取时自动合并字典，环境配置优先级高于公共配置，同名字段自动覆盖
 
#### 5.2 合并示例
 
公共定位器：
 
```yaml
  
search_page:
  search_input: "#kw"
  search_button: "#su"
  result_item: ".result"
``` 
 
测试环境差异化配置：
 
```yaml
  
search_page:
  search_input: "#test-search-input"
``` 
 
最终生效结果：
 
-  `search_input` ：使用环境值  `#test-search-input` （覆盖公共）
-  `search_button` 、 `result_item` ：复用公共值
 
#### 5.3 日志埋点
 
合并后会打印日志，可直观判断是否生效：
 
- 有差异： `页面[search_page]定位器合并完成，环境覆盖1个字段` 
- 无差异： `页面[search_page]无环境特殊定位器，复用公共配置` 
 
### 6. 新增业务页面标准流程
 
1. 在  `locators.yaml`  新增公共定位器节点，写入全环境通用的元素选择器
2. 在各环境  `env/xxx.yaml`  中新增对应页面的URL，如有定位器差异则补充覆盖项
3. 在  `pages/`  下新建对应业务页面类，继承 BasePage，调用  `get_page_locators`  读取定位器
4. 在  `tests/`  下新建对应测试用例文件，调用业务方法编写断言
 
### 7. 兼容性说明
 
- 向下兼容：不传  `--env`  参数默认走 dev 环境，原有执行命令完全可用
- 无破坏性：原有重试、截图、日志、Allure等全部功能不受影响
- 可扩展：新增环境只需新增对应 yaml 文件，无需修改框架代码

## 五、YAML数据驱动参数化体系
 
### 1. 核心设计思想
 
将测试逻辑与测试数据完全解耦：同一套用例代码（打开页面→搜索→断言），通过读取外部 YAML 数据文件批量执行多组测试场景，避免重复编写结构完全相同的用例。新增测试场景仅需追加 YAML 数据，无需修改 Python 代码，大幅降低用例膨胀后的维护成本。
 
### 2. 目录与文件规范
 
- 统一存放路径：项目根目录  `data/`  文件夹，与  `config/` 、 `pages/`  同级
​
- 文件命名规则：与业务页面对应，如  `search_data.yaml`  对应搜索页面测试数据
​
- 数据结构标准：列表套字典，每组数据必含  `case_id` （英文短标识）、 `case_name` （中文场景名）、输入参数、预期校验项
​
- 示例结构：
```yaml
search_cases:
  - case_id: normal_ui_auto
    case_name: "正常场景：UI自动化技术关键词搜索"
    keyword: "Playwright UI自动化测试"
    expect_words: ["playwright", "ui"]
```

### 3. 分层标识工程方案（本项目标准）
 
采用双字段分层设计，同时兼顾兼容性与可读性：
 
| 字段     | 用途                                     | 输出通道                                   | 语言     |
|----------|------------------------------------------|--------------------------------------------|----------|
| case_id  | 供给 parametrize 的 ids 参数、用例筛选  | pytest 终端控制台、命令行筛选              | 英文短标识 |
| case_name | 供给 Allure 动态标题、运行日志展示       | Allure 报告、日志文件、控制台日志          | 完整中文  |

 
三层输出编码边界：
 
1. pytest 终端 ids 行：仅英文，无转义，跨平台零乱码风险
​
2. logging 日志（控制台+落地文件）：原生支持中文，正常展示
​
3. Allure HTML 报告：原生兼容中文，动态标题完整显示业务场景名
 
### 4. 兼容边界说明
 
- ✅ 仅function级单用例隔离范式支持参数化批量执行，每条用例独立环境，无数据污染
​
- ❌ class级夹具范式无法适配数据驱动，多组数据复用同一页面，状态持续累积会导致断言失真

### 5. 用例标签动态绑定
 
测试数据支持通过 `marks` 字段为单组用例独立绑定pytest标签，实现数据与标签统一管理：
 
- 字段格式： `marks: ["smoke", "regression"]` ，支持同时绑定多个标签
​
- 未配置时默认自动打上 `regression` 标签，向下兼容
​
- 与 `-m` 命令行筛选逻辑完全兼容，效果等同于硬编码装饰器
​
- 价值：核心用例标记冒烟集、边界用例仅归入回归，无需修改Python代码

# 通过case_id精准执行单组场景
pytest tests/test_search_parametrize.py::test_search_parametrize[normal_ui_auto] -v

##### 环境变量占位符底层实现说明
方法定位：类私有工具方法 `_replace_env_var`，仅在内部点式取值完成后调用，对外透明，业务代码无需感知。

1. **类型前置判断**
非字符串类型（数字、布尔、列表、字典）直接返回原值，不执行正则处理，避免无意义运算，提升性能。

2. **正则匹配规则**
正则表达式：`r"\$\{([\w-]+)\}"`
- `r`：原始字符串标识，避免反斜杠转义冲突
- `\$`：匹配字面量`$`符号（$在正则中为特殊字符，必须转义）
- `\{`：匹配左大括号`{`
- `([\w-]+)`：捕获分组，匹配变量名，支持字母、数字、下划线、横杠，符合主流系统环境变量命名规范
- `\}`：匹配右大括号`}`
- 匹配目标格式：`${USERNAME}`、`${DB_PWD}`、`${TEST-ACCOUNT}`

3. **替换回调逻辑**
通过 `re.sub` 的函数回调模式实现动态替换：
- 每次匹配到占位符时，自动触发回调函数，提取分组1的变量名
- 调用 `os.environ.get(var_name, default)` 读取环境变量
  - 第1参数：环境变量名称
  - 第2参数：变量不存在时的兜底值，此处返回完整占位符原文，不抛异常
- 环境变量存在则返回真实值，不存在则保留原占位符，便于排查漏配问题

4. **全局替换执行**
调用 `re.sub(pattern, repl, string)` 完成全量替换：
- `pattern`：正则匹配规则
- `repl`：替换回调函数
- `string`：待处理的原始配置字符串
- 默认全局匹配所有占位符，不限次数，支持单条配置中包含多个占位符

5. **调用时机设计**
统一在 `_get_by_dot_path` 方法的取值出口执行替换，一处改造全配置生效：
- 覆盖公共配置、环境配置、测试数据三类读取入口
- 缓存存储原始配置文本，替换仅在每次读取时动态执行，不污染缓存
- 敏感信息不会写入缓存字典，兼顾读取性能与配置安全性

- **使用规范**：所有可变参数禁止硬编码，统一通过本工具类从YAML文件读取

- 配置释义：
​
1.  `testpaths = tests` ：限定仅扫描 `tests` 目录，提升检索速度，规范用例存放
​
2.  `addopts = --ignore=tests/draft -v` ：默认忽略草稿目录，开启详细输出
​
3.  `python_files/python_classes/python_functions` ：固化用例文件、类、函数的命名识别规则
​
4. markers 标签配置说明
- 作用：注册自定义用例标记，配合`-m`参数实现用例分级筛选，同时避免未注册标记抛出告警
- 当前注册三类标签：
  - `smoke`：冒烟测试用例，覆盖核心主流程，用于版本发布前快速验证
  - `regression`：回归测试用例，覆盖全量业务场景，用于迭代后全量验证
  - `debug`：调试专用用例，正式执行默认排除，仅用于本地临时调试
- 查看命令：`pytest --markers` 可查看所有已注册标记及说明

- 强制编码规范（红线）：
​
1. 禁止写入中文、全角符号注释：底层依赖 `iniconfig` 库读取，Windows环境默认采用GBK解码，中文会触发 `UnicodeDecodeError` 解码崩溃；且ini标准不支持内部声明编码，无法通过加配置项解决
​
2. 所有配置说明统一维护在本文档中，ini文件仅保留纯英文配置
​
3.  `addopts` 内禁止写入 `--reruns` / `--reruns-delay` 参数，否则会被识别为用户手动传参，覆盖 `config.yaml` 中的全局重试配置

#### 6.1 三层配置体系分层说明
 
框架存在三类完全独立的配置载体，分属不同运行层级，职责隔离互不耦合，修改时需区分场景：
 
| 配置文件     | 所属层级         | 读取主体                 | 核心职责                                                                 | 关键注意事项                                                                                                                                 |
|--------------|------------------|--------------------------|--------------------------------------------------------------------------|----------------------------------------------------------------------------------------------------------------------------------------------|
| pytest.ini   | Pytest原生框架层 | pytest内核自动读取       | 控制用例检索规则、默认执行命令参数、用例命名规范、目录过滤                 | 1. addopts内参数等价手动命令行输入，会被 config.getoption() 捕获<br>2. 禁止在ini固化 --reruns/--reruns-delay，否则会永久覆盖config.yaml全局重试配置<br>3. 仅存放静态固定执行规则，不存放业务动态参数 |
| conftest.py  | 自定义扩展逻辑层 | pytest插件钩子自动加载   | 注册自定义命令行参数、全局夹具、失败截图钩子、初始化环境标识、注入Allure元数据 | 无配置存储能力，仅承载代码逻辑，不存放参数常量                                                                                                |
| config.yaml  | 业务自定义配置层 | 自研YamlReader工具读取   | 浏览器、超时、视口、报告、重试策略等业务动态可变参数；多环境差异化URL/定位器 | 完全独立于pytest原生体系，不受pytest.ini参数影响，环境切换、重试默认值均由此文件驱动                                                          |


关键联动逻辑：
 
1. 重试机制： `pytest.ini`  的  `addopts`  若写入 `--reruns` ，会被识别为用户主动传参，触发优先级规则：命令行参数 > config.yaml全局配置，YAML重试配置直接失效；本项目默认未加入重试相关参数，不会破坏优先级。
​
2. 多环境切换：完全由 YamlReader 自主读取 env 目录文件，与pytest原生配置无关， `pytest.ini`  不会产生任何干扰。

#### 6.2 目录过滤选型规范
 
| 配置项          | 过滤粒度             | 适用场景                                                     |
|-----------------|----------------------|--------------------------------------------------------------|
| norecursedirs   | 仅支持完整文件夹     | 需要彻底跳过整个草稿、临时目录，写法简洁                     |
| --ignore=xxx    | 支持目录/单个py文件  | 同目录仅屏蔽某一条废弃脚本，其余用例正常执行                 |
 
> pytest.ini 完整配置释义、编码红线见第七章7.6.2。

## 十、常见问题排查
 
### 1. 提示找不到浏览器
 
- 确认 Edge 路径填写正确，可在 Edge 地址栏输入  `edge://version`  查看执行路径
- 替换为  `channel="msedge"`  自动识别系统 Edge
 
### 2. 夹具不生效/提示找不到page
 
- 确认 conftest.py 放在测试文件同级目录
- 确认测试函数参数名与夹具函数名完全一致
 
### 3. 元素定位失败/操作超时
 
- 优先使用 ID、CSS 选择器定位，减少层级依赖
- 区分两种场景：DOM不存在则加长等待时间；DOM存在但不可见，可使用  `state="attached"`  先等待挂载
- 检查是否有弹窗、遮罩遮挡目标元素，补充对应关闭逻辑
 
### 4. 输入框fill操作超时（元素存在但不可见）
 
- 根因： `page.fill()`  内置强制校验，要求元素必须可见、可启用、可编辑，CSS隐藏的元素无法通过该校验
- 优化方案：采用「原生等待+JS兜底」双层容错逻辑，极端场景自动降级注入
 
### 5. 断言偶发失败
 
- 优先排查是否为页面未加载完成就读取文本，替换固定等待为元素级智能等待
- 若为排版/大小写差异导致，切换为多关键词+统一小写的断言方案
 
### 6. YAML配置读取报错
 
-  `KeyError` ：根据异常提示的路径与字段，检查 yaml 文件中对应层级的字段名是否拼写正确
- 文件找不到：确认  `config/`  目录在项目根目录，文件名与代码中读取的名称完全一致
-  `ModuleNotFoundError: No module named 'yaml'` ：执行  `pip install pyyaml == 6.0.3` 或 `pip install -r requirements.txt` 安装依赖.

### 7. Allure命令找不到
 
- 确认Allure CLI已解压，且bin目录已加入系统Path
​
- 确认JDK17配置正常， `java -version` 可正常输出版本
​
- 修改环境变量后必须重启终端
 
### 8. Allure报告不显示截图
 
- 确认用例是call阶段失败，setup/teardown阶段失败不触发截图
​
- 确认  `config.yaml`  中  `auto_screenshot`  为true
​
- 确认测试用例注入了 `page` 夹具 （类级自定义PO夹具不兼容）
 
### 9. pytest高版本运行报错（AttributeError: 'str' object has no attribute ...）
 
- 根因：pytest 8.x/9.x 对内部节点API做了破坏性变更，allure-pytest 2.13.2等老插件不兼容
- 解决方案：锁定 pytest==7.4.4 企业稳定版，执行  `pip install pytest==7.4.4`  降级即可
- 选型说明：7.x版本插件生态最完整，是企业级自动化项目的主流选型

### 10. 日志重复打印两行
 
- 根因：子logger重复挂载控制台Handler + 日志向上传播，导致同一条日志被两个Handler各打印一次
- .\common\logger.py错误代码示例：
```python
    if not logger.handlers:
        logger.addHandler(console_handler)
    会导致如下输出(节选)
    2026-06-26 22:50:03 - yaml_reader - INFO - 读取配置节点，文件：config.yaml，路径：timeout.element_wait
    2026-06-26 22:50:03 - yaml_reader - INFO - 读取配置节点，文件：config.yaml，路径：timeout.element_wait
```
- 解决方案：使用框架内置logger.py，子logger仅通过传播复用根logger的Handler，不重复添加 

### 11. 重试机制不生效
 
- 检查  `config.yaml`  中  `rerun.enable`  是否为 true
- 确认命令行没有传入  `--reruns 0`  显式关闭重试
- 确认用例是call阶段失败，setup/teardown阶段失败不触发重试
- 检查 pytest-rerunfailures 插件是否正确安装
 
### 12. 环境切换不生效
 
- 检查  `--env`  参数值是否在 choices 白名单内
- 确认对应环境的 yaml 文件存在于  `config/env/`  目录
- 查看启动日志是否打印  `全局运行环境已设置为：xxx` 

### 13. 启动报错UnicodeDecodeError（pytest.ini解码失败）
 
- 根因：pytest.ini 底层 iniconfig 库在Windows下默认GBK解码，中文注释会触发解码崩溃
​
- 解决方案：严格遵守规范，pytest.ini禁止写入中文注释，所有说明统一维护在本文档
 
### 14. 类夹具用例失败不触发自动截图
 
- 根因：失败截图钩子依赖标准 `page` 夹具参数，自定义class级PO夹具无该变量
​
- 解决方案：正式业务用例统一使用function级page夹具范式；类演示用例不依赖自动截图能力

### 15. 参数化用例终端显示\u开头unicode转义字符
 
- 根因：pytest 对 ids 参数默认采用 ASCII 安全编码，中文强制转义，属于框架底层行为，无法通过编码配置根治
​
- 解决方案：采用  `case_id`  英文短标识作为 ids，中文场景名通过日志、Allure 标题展示，从根源规避转义问题

### 16. Locator报 strict mode violation 严格模式错误
- 根因：Playwright的Locator默认开启严格模式，执行wait_for、click等单元素操作时，若选择器匹配到多个DOM元素会直接抛出异常；动态渲染页面在等待过程中新增元素也可能触发该校验
- 避坑说明：`.first()`仅做后置过滤，等待过程中元素动态新增仍可能触发严格模式校验；`.nth(下标)`直接锁定指定索引元素，稳定性更高
- 标准解决方案：
  1. 限定父级唯一容器，缩小选择器匹配范围
  2. 使用`:not()`过滤广告、推荐等无关条目，减少匹配数量
  3. 末尾搭配`.nth(0)`锁定目标单元素，从根源规避多匹配报错

### 17. pytest --markers 标签说明中文乱码（Windows）
- 根因：Windows 终端默认采用 GBK 编码，pytest 输出的中文标记描述解码异常
- 解决方案：执行查看命令前，先在终端运行 `chcp 65001` 切换为 UTF-8 编码，再执行查看命令
- 说明：该操作仅当前终端会话生效，不修改系统全局配置；Mac/Linux系统无此问题

### 18. 多进程并发执行时截图文件被覆盖
- 根因：多个子进程同时执行失败截图，若文件名规则相同会出现同名文件覆盖
- 解决方案：框架已内置进程标识拼接逻辑，并发场景自动在文件名前增加gw0/gw1等进程前缀
- 验证方式：并发执行后查看截图目录，文件名前缀应包含不同的进程标识


## 十一、Allure可视化报告体系
 
### 1. 双报告对比

| 维度       | pytest-html                | Allure                                          |
|------------|----------------------------|-------------------------------------------------|
| 环境依赖   | 无，纯Python插件           | 需要JDK + Allure CLI                            |
| 报告形式   | 单HTML文件                 | 交互式网页，支持筛选、分类                      |
| 步骤展示   | 无分层                     | 支持step嵌套分层展示                            |
| 附件能力   | 弱                         | 支持截图、文本、HTML等多类型附件                |
| 适用场景   | 快速调试、本地验证         | 正式交付、团队协作、问题追溯                    |

### 2. Allure装饰器分层规范
 
采用三级标签体系，实现报告按业务模块分组展示：
 
1.  `@allure.feature("模块名")` ：一级大分类，对应业务模块
​
2.  `@allure.story("场景名")` ：二级细分，对应模块下的业务场景
​
3.  `@allure.severity(级别)` ：用例严重等级，可选：BLOCKER/CRITICAL/NORMAL/MINOR/TRIVIAL
​
4.  `@allure.title("用例名称")` ：自定义中文用例名称，替代函数名
​
5.  `@allure.step("步骤描述")` ：操作步骤标记，支持参数占位，形成嵌套步骤树
 
执行逻辑：运行  `pytest --alluredir`  时，allure-pytest插件自动扫描装饰器，将标签、步骤、附件信息序列化存入json文件，Allure CLI渲染时读取并生成可视化页面。
 
### 3. 失败截图钩子底层原理
 
#### 3.1 包装型钩子机制
 
`pytest_runtest_makereport`  是pytest内置钩子，函数名必须与官方固定名称完全一致才能被识别。
 
- 普通钩子：与原生逻辑并行执行，无法获取用例最终结果
​
- 包装型钩子（ `hookwrapper=True` ）：套在原生钩子外层，先暂停执行，等原生逻辑跑完拿到结果后，再执行自定义逻辑，适合失败截图、日志记录等后置处理

#### 3.2 钩子yield执行时序

```plantext
【外层：自定义包装钩子函数】
    yield前：用例未执行，可做前置埋点、计时、日志打印
    outcome = yield  # 交出控制权
        └── 【内层：pytest底层原生钩子逻辑】
            生成报告结果对象，标记用例阶段、成功/失败、异常信息
    yield后：获取执行结果，判断失败则执行截图、挂载附件逻辑
```

#### 3.3 代码中三处yield的区分
 
项目中共有3处 `yield` ，分属两套不同机制，不可混淆：
 
| 位置        | 类型        | 核心作用                                                 | 触发时机                          |
|-------------|-------------|----------------------------------------------------------|-----------------------------------|
| browser夹具 | 夹具yield   | 分割浏览器初始化与销毁，返回浏览器实例                     | 整个测试会话仅执行1次              |
| page夹具    | 夹具yield   | 分割页面创建与关闭，返回页面对象，实现用例数据隔离         | 每条测试用例执行1次               |
| 报告钩子    | 钩子yield   | 等待原生钩子执行完成，获取用例运行结果做截图等二次后置处理 | 每条用例触发3次（setup/call/teardown） |

 
> 整体执行洋葱模型：从外到内依次执行前置（browser→page→钩子前置→用例），从内到外依次执行后置（用例→钩子后置→page→browser）。
 
#### 3.4 钩子参数说明
 
-  `item` ：当前测试用例对象，用于获取用例名称、夹具实例
​
-  `_call` ：钩子强制固定入参，包含阶段详细信息；包装钩子可通过 `outcome` 获取等价信息，未直接使用故加下划线标识，不可删除，否则钩子失效

- 兼容边界：仅注入标准 `page` 夹具的用例支持失败自动截图，自定义类级PO夹具不兼容

#### 3.5 报告元数据自动生成
 
框架在 `pytest_configure` 钩子中自动生成两份Allure标准元数据文件，无需手动操作：
 
1. `environment.properties`
​
- 格式：标准Java Properties格式， `键=值` ，每行一条
​
- 内容：运行环境、浏览器、无头模式、视口尺寸、重试配置、Python/Pytest版本
​
- 效果：Allure报告首页Environment面板自动展示，快速确认运行上下文

​

- 效果：补全报告Executors面板信息，适配本地/CI不同执行场景

## 十二、标准化日志体系
 
### 1. 设计目标
 
实现全链路操作可追溯，兼顾控制台可读性与文件排查完整性，支持按日期归档，避免日志重复、冗余。
 
### 2. 核心能力
 
文件路径： `common/logger.py` 
 
- 双输出通道：日志同时输出到控制台 + 本地日志文件
- 按日期拆分：每天自动生成独立日志文件，命名格式  年-月-日.log 
- 分级输出：INFO/WARNING/ERROR三级分级
- 防重复机制：子logger通过传播复用根logger处理器
- 自动建目录：首次运行自动创建  `logs/`  文件夹

> 工具类完整API与使用规范见第七章7.1.4。
 
### 3. 日志分级规范
 
| 级别   | 使用场景                                   | 示例                                                     |
|--------|--------------------------------------------|----------------------------------------------------------|
| INFO   | 正常流程节点、业务操作启动/完成             | 浏览器启动、页面打开、元素等待完成                       |
| WARNING| 预期内降级、非致命异常                     | 原生操作失败切换JS兜底、未匹配到弹窗                     |
| ERROR  | 底层操作失败、真正故障                     | 元素等待超时、配置文件不存在                             |
 
### 4. 异常输出策略
 
采用「底层保留完整详情、上层精简输出」的分层原则：
 
- 底层工具方法（如 `wait_for_element` ）：保留完整异常信息 `str(e)` ，包含Playwright原生调用日志，用于深度排查
- 上层业务封装（如 `safe_fill` / `safe_click` ）：仅打印异常类型，保持控制台整洁
- 堆栈信息：默认不打印到控制台，需要详细排查时查看日志文件

### 5. 日志目录说明
 
- 运行后自动生成项目根目录下  `logs/`  文件夹
- 日志文件按天追加，不会覆盖历史记录
- 建议将  `logs/`  加入  `.gitignore` ，不提交到代码仓库

## 十三、公共断言工具集

### 1. 设计目标
将UI自动化高频断言逻辑统一收敛，统一日志埋点、失败提示格式，避免用例层重复编写判断代码，提升断言一致性与可维护性。

### 2. 工具类说明
文件路径：`common/assert_utils.py`
采用静态工具类设计，无需实例化，直接通过类名调用，覆盖四类核心场景：

| 方法名              | 作用                               | 适用场景                     |
|---------------------|------------------------------------|------------------------------|
| assert_any_contains | 断言文本包含任意一个目标关键词     | 搜索结果、列表内容模糊匹配   |
| assert_all_contains | 断言文本包含全部目标关键词         | 完整文案、详情页精准校验     |
| assert_equal        | 断言两个值完全相等                 | 标题、数量、状态值精准校验   |
| assert_not_empty    | 断言对象非空                       | 列表、返回结果、文本非空校验 |

### 3. 使用示例
```python
from common.assert_utils import AssertUtils

# 多关键词任意匹配
AssertUtils.assert_any_contains(result_text, ["playwright", "ui"], desc="搜索结果校验")

# 值相等校验
AssertUtils.assert_equal(page.title(), "百度一下，你就知道", desc="页面标题校验")
```

### 4. 使用规范
 
- 存量用例保留原生 `assert` 写法，无需强制改造，完全兼容
​
- 新增测试用例、新增断言逻辑优先使用工具类，逐步统一规范
​
- 复杂业务断言可在工具类中扩展对应方法，不建议在用例中编写复杂判断逻辑

> 完整方法说明与注意事项见第七章7.1.3。

## 十四、用例标签分级体系

### 1. 设计目标
通过标签对用例进行分级归类，实现不同执行场景下的快速筛选，避免全量执行耗时过长，适配日常调试、版本冒烟、全量回归不同诉求。

### 2. 三级标签定义

| 标签名     | 定位     | 适用场景                 | 包含用例 |
|------------|----------|--------------------------|----------|
| smoke      | 冒烟测试 | 版本发布前快速验证核心主流程 | 核心链路单用例，覆盖最基础业务能力 |
| regression | 回归测试 | 迭代上线前全量业务验证 | 所有正式业务用例，覆盖正常、边界、异常场景 |
| debug      | 调试专用 | 本地临时调试、废弃验证脚本 | 调试中、临时验证、待废弃的用例 |

### 3. 标签使用规范
- 核心主流程用例建议同时标记 `smoke + regression`，既参与冒烟也参与全量回归
- 参数化批量用例、边界异常用例仅标记 `regression`
- 临时调试、草稿类用例标记 `debug`，正式回归执行时统一排除
- 新增用例必须根据业务重要性标记对应标签，禁止无标签用例混入正式执行集

### 4. 标签筛选语法
通过 `-m` 参数搭配逻辑运算符组合筛选：
- 单个标签：`-m smoke` 仅执行冒烟用例
- 同时满足：`-m "smoke and regression"` 执行同时有两个标签的用例
- 满足其一：`-m "smoke or debug"` 执行满足任意一个标签的用例
- 排除标签：`-m "not debug"` 排除调试用例
- 组合逻辑：`-m "regression and not debug"` 执行正式回归用例，排除调试脚本

### 5. Windows终端中文乱码处理
Windows 终端默认 GBK 编码，执行 `pytest --markers` 时标签中文描述可能出现乱码，解决方案：
1. 执行查看命令前，先在终端运行：`chcp 65001`
2. 再执行：`pytest --markers` 即可正常显示中文说明
> 说明：`chcp 65001` 仅对当前终端会话生效，关闭窗口后自动恢复系统默认编码，不会修改全局系统配置；Mac / Linux 终端默认 UTF-8 编码，无需此操作。

## 十五、多进程并发执行体系

### 1. 设计目标
基于 pytest-xdist 插件实现用例多进程分发执行，充分利用多核CPU资源，大幅缩短全量回归测试的执行时长，提升迭代效率。

### 2. 核心改造点
#### 2.1 并发执行原理
- 主进程（master）负责收集测试用例，按照分发策略分配给多个子进程（worker）
- 每个子进程独立运行测试用例，各自维护独立的浏览器实例、夹具生命周期，用例数据完全隔离
- 主进程汇总所有子进程的执行结果，生成统一的测试报告

#### 2.2 截图并发兼容设计
多进程同时执行失败截图时，若文件名仅包含用例名与时间戳，会出现同名文件相互覆盖的问题。解决方案：
1. 读取 pytest-xdist 自动注入的环境变量 `PYTEST_XDIST_WORKER` 获取当前进程标识（gw0、gw1、gw2...）
2. 将进程标识拼接到截图文件名最前端，保证每个进程生成的截图文件名唯一
3. 单进程执行时该环境变量不存在，使用 `master` 作为兜底标识
   - 规避 `None` 前缀的不规范文件名
   - 直观区分截图是串行执行还是并发执行生成
   - 保证文件名永远为字符串类型，避免格式化异常

#### 2.3 并发分发模式
| 分发模式 | 说明 | 适用场景 |
|----------|------|----------|
| load | 负载均衡模式，动态将待执行用例分配给空闲进程 | 通用场景，默认推荐 |
| loadfile | 按文件分组，同一测试文件的用例分配给同一个进程 | 用例文件内存在依赖、需共享夹具的场景 |
| loadscope | 按作用域分组，同一会话级夹具的用例分配给同一进程 | 存在重量级session级夹具的场景 |

### 3. 注意事项
- 并发执行时每个进程独立启动浏览器，窗口数量与进程数一致，属于正常现象
- class级共享夹具用例在并发场景下可能出现状态污染，正式回归推荐使用function级隔离用例
- 进程数并非越多越好，建议设置为CPU物理核心数，过多进程会因资源争抢导致执行变慢

## 十六、完整代码逻辑说明
### 16.1 整体执行全链路
从输入命令到生成报告，完整执行流程分为6个阶段，按顺序执行：

#### 阶段1：启动初始化（全局仅1次）
1.  读取`pytest.ini`，加载用例扫描规则、默认执行参数、自定义标记
2.  触发`pytest_addoption`钩子，注册`--env`自定义命令行参数
3.  触发`pytest_configure`生命周期钩子，完成三件事：
    - 调用`YamlReader.set_current_env()`初始化全局环境标识，后续所有配置读取自动对齐
    - 读取`config.yaml`的重试配置，用户未手动传参时注入全局重试规则
    - 生成Allure报告配套的`environment.properties`与`executor.json`元数据文件

#### 阶段2：用例收集
1.  按照`testpaths`限定的目录扫描测试文件，识别符合命名规范的测试函数
2.  解析用例上的`@pytest.mark`标签、参数化配置，生成待执行用例列表
3.  开启多进程时，主进程将用例按分发策略分配给多个worker子进程

#### 阶段3：会话级夹具初始化（全局仅1次）
1.  第一条用例执行前，触发`browser`会话级夹具
2.  读取`config.yaml`浏览器配置，校验本地exe路径是否存在
3.  路径有效则通过指定路径启动Edge，路径无效自动降级为`channel="msedge"`自动识别
4.  浏览器实例创建完成后挂起，全程所有用例共享同一个浏览器进程

#### 阶段4：单条用例执行（每条用例重复执行）
1.  触发`context`函数级夹具：创建独立浏览器上下文，设置视口、注入反爬JS、锁定窗口尺寸
2.  触发`page`函数级夹具：新建独立标签页，完成初始化后交付给测试用例
3.  执行测试用例业务代码：
    - 实例化业务PO类，PO类继承`BasePage`，初始化时从YAML加载定位器与环境URL
    - 调用PO业务方法，底层操作全部走`BasePage`的三层容错逻辑
    - 执行断言，支持原生断言与`AssertUtils`工具类两种方式
4.  用例执行完毕，依次销毁`page`、`context`夹具，关闭标签页与上下文，隔离下一条用例

#### 阶段5：结果后置处理
1.  每条用例三个阶段（setup/call/teardown）都会触发`pytest_runtest_makereport`包装钩子
2.  仅`call`阶段失败时，触发自动截图逻辑：
    - 读取报告配置，校验自动截图开关
    - 获取当前用例的`page`实例，截取全页面保存到本地目录
    - 将截图文件挂载到Allure报告附件中
3.  开启重试时，失败用例自动重建页面重跑，每轮重试独立截图、独立记录

#### 阶段6：收尾销毁
1.  所有用例执行完毕，回到`browser`夹具后置逻辑，关闭浏览器进程
2.  释放Playwright驱动资源
3.  生成最终执行统计结果，Allure原始数据全部写入指定目录

### 16.2 核心分层调用关系
框架采用自上而下的四层调用架构，上层依赖下层，下层不感知上层，单向依赖无循环：

| 层级         | 代表文件       | 核心职责                                   | 调用方向             |
|--------------|----------------|--------------------------------------------|----------------------|
| 测试用例层   | `tests/*.py`   | 业务场景编排、断言校验                     | 调用业务PO层         |
| 业务PO层     | `pages/*.py`   | 页面元素定位、业务流程封装                 | 调用公共工具层       |
| 公共工具层   | `common/*.py`  | 通用操作、配置读取、日志、断言工具         | 读取配置文件层       |
| 配置文件层   | `config/*.yaml`| 所有可变参数、定位器、环境数据             | 被上层读取，无代码逻辑 |


调用链路示例（搜索用例）：
`test_search_parametrize.py` → `SearchPage.search()` → `BasePage.safe_fill()/safe_click()` → `YamlReader`读取配置 → 底层Playwright API执行操作

### 16.3 关键机制联动逻辑
#### 1. 环境切换联动
- 命令行`--env`参数 → `pytest_configure`设置全局环境标识
- 所有`YamlReader`读取环境配置、定位器合并时，自动使用当前环境
- 业务代码无需手动传环境参数，全局统一生效

#### 2. 重试机制联动
- 优先级规则：单用例装饰器 > 命令行参数 > YAML全局配置
- 每轮重试都会重新执行`page`夹具，新建独立页面，无状态污染
- 每轮失败都会触发截图钩子，生成独立截图文件，可追溯每轮失败现场

#### 3. 失败截图联动
- 依赖标准`page`函数级夹具，通过`item.funcargs["page"]`获取页面对象
- 仅处理业务执行阶段失败，夹具初始化/销毁异常不触发无效截图
- 多进程场景自动拼接进程标识前缀，避免文件覆盖

#### 4. Locator与JS兜底联动
- 业务PO层预生成Locator实例，同时缓存原始选择器字符串
- 调用`safe_fill/safe_click`时，原生操作使用Locator对象
- 降级到JS兜底时，使用缓存的原始选择器，规避Locator私有属性不可读的限制

### 16.4 新增业务页面标准开发流程
按照框架规范，新增一个业务页面遵循以下4步，所有扩展均符合分层解耦原则：
1.  **配置层**：在`locators.yaml`新增页面定位器，在各环境yaml中补充页面URL与差异化配置
2.  **PO层**：在`pages/`下新建页面类，继承`BasePage`，封装业务流程方法
3.  **数据层**：如需参数化，在`data/`下新增对应测试数据文件，配置`marks`标签
4.  **用例层**：在`tests/`下新建测试文件，调用PO方法编写断言，打上对应用例标签
